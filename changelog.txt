V2.1 
This is a massive change..
Rewritten the receiver routine to make it more efficient. Receive mode should now be able to handle far more incoming streams for a given set of hardware
Traceroute has been rewritten for OSX/linux - much less CPU intensive
A Raspberry Pi 3 can now happily generate 30Mbps all day without cooking
The 'tx bps' counter on the Transmitter now gives you the 'actual tx bps' (as opposed to the rate you specified) so you know what you're sending. It also puts up a warning if the tx rate is exceeding what you specified at the outset. This gives you some warning that you're probably overloading the hardware.
Packet loss is now to two decimal places
Adds more information to the report - the test duration, the total bytes received, detail about route changes etc.
Also records the TTL value of the incoming packets. This allows a secondary source info about routing changes.
Additional Event types for the report: Traceroute change, Received IP TTL, Source address/port, duplicate packets received, out of order packets received
Help page shows the filenames/paths being saved to
'Burst Mode'  - This will temporarily double the TX rate for a period of 5 secs to allow testing of Mikrotik queues to see what happens when there is a sudden increase in the amount of transmitted data
V2.2 Rewritten traceroute. Shared code between Windows and Linux/OSX - much easier to maintain. Better error handling for links where the traceroute fails (because the routers in the way let through packets with ttls already decremented to zero. 
Adds remote control of transmitter from receiver
Coded added to allow listening on multiple ports at once
V2.3 Pastebin is done away with. Now, if the local clipboard copy fails (when generating a report), isptest will launch 'less' as a subprocess to allow the full report text to be copied
V2.4 
* Traceroute on Linux/OSX now uses ‘actual source port and destination of tx stream’ so a valid description of the path taken (on Windows/Scapy, still uses an OS generated source port)
* Adds traceroute last updated timestamp (with stable routes) on Receiver
* adds a StreamResumed Event. This calculates the time diff between a StreamLost Event being detected and then the stream resuming. It's required because some outages exhibit a property whereby the packets at the start/end of a glitch get squeezed in time by the link. This leads to an erroneous calculation of glitch duratiomn (which has hithertoo been calculated using the timestamps of the packets either side of the glitch
* An ‘Expired Stream’ can be resurrected 
* Traceroute History viewer (Less/clipboard)
* Configurable event display filter options (filter by Glitch, Routing change etc.)
* Rewritten route change detection code
* Reverse order of traceroute events in export to clipboard
